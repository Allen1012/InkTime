# InkTime 数据存储设计

## 1. 存储架构概述

InkTime 项目采用分层存储架构，结合数据库存储和文件系统存储，实现照片元数据和文件的高效管理。

```
+----------------------------------+
|        存储架构                    |
+----------------------------------+
|                                  |
|  +----------------------------+   |
|  |  数据存储层                  |   |
|  +----------------------------+   |
|  |                            |   |
|  |  +----------------------+   |   |
|  |  |  SQLite 数据库        |   |   |
|  |  +----------------------+   |   |
|  |  |  photo_scores.db     |   |   |
|  |  +----------------------+   |   |
|  |                            |   |
|  |  +----------------------+   |   |
|  |  |  文件存储              |   |   |
|  |  +----------------------+   |   |
|  |  |  照片目录: data/photos/  |   |   |
|  |  |  输出目录: data/output/ |   |   |
|  |  +----------------------+   |   |
|  +----------------------------+   |
|                                  |
+----------------------------------+
```

## 2. 数据库设计

### 2.1 数据库选择

**选择 SQLite 的原因：**
- **轻量级**：无需独立服务器进程，以单个文件形式存在
- **零配置**：不需要安装和配置数据库服务
- **嵌入式**：直接集成到 Python 应用中，使用标准库 `sqlite3`
- **适合小型应用**：项目规模适中，照片数量通常在几千张以内
- **易于部署**：无需额外的数据库服务，简化了部署流程
- **性能足够**：对于单用户、中小规模数据量，SQLite 性能完全满足需求

**与 MySQL 对比：**
- **复杂性**：MySQL 需要单独安装、配置和管理数据库服务
- **资源占用**：MySQL 占用更多系统资源，对于嵌入式或小型服务器不友好
- **部署成本**：增加了部署和维护的复杂度
- **过度设计**：对于 InkTime 这样的项目，MySQL 的功能过于强大，属于过度设计

### 2.4 数据操作

**数据写入**：
- `analyze_photos.py` 分析照片后将结果写入数据库
- 使用事务确保数据一致性
- 支持断点续跑，已处理的照片不会重复分析

**数据读取**：
- `render_daily_photo.py` 读取数据库选择要渲染的照片
- `server.py` 读取数据库提供 WebUI 和 API 数据
- 使用索引优化查询性能

## 3. 文件存储设计

### 3.1 目录结构

```
data/
├── photos/          # 原始照片目录
│   ├── 2023/
│   ├── 2024/
│   └── ...
├── output/          # 输出目录
│   ├── photo_0.bin   # 渲染后的照片文件
│   ├── photo_1.bin   # 渲染后的照片文件
│   ├── latest.bin    # 最新渲染的照片文件
│   └── preview.png   # 预览图
└── photo_scores.db  # SQLite 数据库文件
```

### 3.2 文件命名规范

**原始照片**：保持原始文件名，按年份/月份组织
**渲染文件**：
- `photo_{idx}.bin`：第 idx 张渲染照片
- `latest.bin`：最新渲染的照片
- `preview.png`：渲染效果预览图

### 3.3 文件存储策略

- **照片存储**：原始照片保持不变，存储在 `data/photos/` 目录
- **渲染文件**：每次渲染生成新的 `.bin` 文件，覆盖旧文件
- **预览图**：每次渲染生成新的预览图，覆盖旧文件
- **文件清理**：定期清理过期的渲染文件，节省存储空间

## 4. Docker 容器中的数据存储

### 4.1 数据持久化方案

Docker 容器默认是临时的，容器重启或重建后，内部数据会丢失。因此需要使用持久化存储方案。

**推荐方案：**

1. **绑定挂载（Bind Mounts）**
   - 将主机上的 `data/` 目录挂载到容器内的对应目录
   - 优点：数据直接存储在主机文件系统，易于备份和管理
   - 配置示例：
     ```yaml
     volumes:
       - ./data:/app/data
     ```

2. **Docker 卷（Volumes）**
   - 创建专门的数据卷存储 `data/` 目录
   - 优点：由 Docker 管理，更安全，适合生产环境
   - 配置示例：
     ```yaml
     volumes:
       - inktime_data:/app/data
     volumes:
       inktime_data:
     ```

**不推荐的方案：**
- **存储在容器内**：数据随容器生命周期结束而丢失，不适合持久化存储
- **单独起 SQL 服务**：增加了系统复杂度，违背了项目轻量级设计原则

### 4.2 Docker Compose 配置示例

```yaml
version: '3'
services:
  inktime:
    build: .
    volumes:
      - ./data:/app/data  # 绑定挂载主机目录
    ports:
      - "5005:5005"
    environment:
      - TZ=Asia/Shanghai
```

### 4.3 数据备份策略

- **定期备份**：定期备份 `data/` 目录，包括数据库文件和照片
- **备份方法**：
  - 使用 cron 任务定时备份
  - 使用 Docker 卷快照
  - 手动备份到外部存储
- **恢复策略**：将备份数据复制回 `data/` 目录即可恢复

## 5. 性能优化

### 5.1 数据库优化

- **索引优化**：为常用查询字段创建索引
- **查询优化**：使用参数化查询，避免 SQL 注入
- **事务管理**：批量写入时使用事务，提高性能
- **数据库压缩**：定期使用 VACUUM 命令压缩数据库

### 5.2 文件系统优化

- **目录结构**：按年份/月份组织照片，提高文件系统访问效率
- **文件缓存**：使用操作系统文件缓存，提高文件访问速度
- **存储介质**：使用 SSD 存储，提高读写性能

### 5.3 扩展性考虑

- **数据量增长**：当照片数量达到数万张时，可考虑以下方案：
  - 分库分表：按年份或月份拆分数据库
  - 迁移到 PostgreSQL：支持更大的数据量和并发访问
- **存储扩展**：
  - 使用网络存储（NAS）扩展存储空间
  - 实现照片压缩存储，节省空间

## 6. 安全性考虑

### 6.1 数据安全

- **备份策略**：定期备份数据，防止数据丢失
- **访问控制**：限制对 `data/` 目录的访问权限
- **数据加密**：对敏感照片可考虑加密存储

### 6.2 防止数据损坏

- **文件系统选择**：使用可靠性高的文件系统（如 ext4、NTFS）
- **数据校验**：定期校验文件完整性
- **错误处理**：实现健壮的错误处理，防止数据写入失败

## 7. 未来扩展

### 7.1 存储扩展

- **云存储集成**：支持将照片存储到云服务（如 AWS S3、阿里云 OSS）
- **分布式存储**：对于大型部署，可考虑分布式存储方案

### 7.2 数据库升级

- **PostgreSQL 支持**：提供从 SQLite 迁移到 PostgreSQL 的工具
- **数据库集群**：支持主从复制，提高可用性

### 7.3 数据管理工具

- **数据导入导出**：支持批量导入导出照片和元数据
- **数据清洗**：提供重复照片检测和清理工具
- **数据分析**：提供照片统计和分析工具

## 8. 总结

InkTime 项目采用 SQLite 数据库和文件系统相结合的存储方案，既满足了功能需求，又保持了系统的简洁性和可维护性。在 Docker 容器中，通过绑定挂载或 Docker 卷实现数据持久化，确保数据不会因容器重启而丢失。

这种存储设计具有以下优势：

1. **轻量级**：SQLite 无需额外服务，部署简单
2. **高效**：适合中小规模数据量，查询性能良好
3. **可靠**：数据持久化方案确保数据安全
4. **可扩展**：预留了向更大规模存储方案迁移的空间
5. **易于维护**：简单的存储结构便于管理和备份

通过这种设计，InkTime 项目能够高效管理照片数据，为用户提供流畅的照片浏览和展示体验。
